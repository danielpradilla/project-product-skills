# PRINCE2 Management Products Reference

Use this as a shared reference for the PRINCE2 skills. Tailor the set to the project size, risk, governance model, and local standards.

## Core Baseline Products

| Management Product | Purpose | Typical Owner | Created |
|---|---|---|---|
| Project mandate | Defines the trigger or request for the project. | Corporate, programme, portfolio, or customer | Before startup |
| Project brief | Gives enough information to decide whether to initiate the project. | Project manager | Starting Up a Project |
| Business case | Explains why the project is justified. | Executive | Startup, then maintained throughout |
| Project initiation documentation | Baseline definition, controls, approaches, plan, and justification. | Project manager | Initiating a Project |
| Project product description | Describes the final product and acceptance criteria. | Project manager with senior user input | Startup/initiation |
| Product description | Defines an individual product, its quality criteria, and acceptance method. | Project manager or team manager | Initiation/stage planning |

## Plans And Control Products

| Management Product | Purpose | Typical Owner | Created |
|---|---|---|---|
| Project plan | Forecast for the whole project. | Project manager | Initiating a Project |
| Stage plan | Plan for a management stage. | Project manager | Initiation or stage boundary |
| Team plan | Team-level delivery plan if useful. | Team manager | Managing Product Delivery |
| Exception plan | Replacement plan after tolerance breach. | Project manager | Exception handling |
| Work package | Agreement between project manager and team manager for product delivery. | Project manager/team manager | Controlling a Stage |

## Approaches

| Management Product | Purpose |
|---|---|
| Benefits management approach or benefits review plan | Defines how benefits will be measured and owned after delivery. |
| Communication management approach | Defines who needs what information, when, and through which channel. |
| Quality management approach | Defines quality standards, methods, responsibilities, and records. |
| Risk management approach | Defines risk scales, appetite, roles, reporting, and escalation. |
| Issue and change management approach | Defines how issues, requests for change, and off-specifications are controlled. |
| Commercial management approach | Defines supplier and contract control when external delivery is involved. |
| Data or information management approach | Defines information governance, storage, access, retention, and data responsibilities. |
| Sustainability management approach | Defines sustainability objectives, measures, controls, and reporting where relevant. |

## Logs, Registers, And Records

| Management Product | Purpose |
|---|---|
| Daily log | Informal notes, early risks/issues, and project manager working record. |
| Risk register | Formal risk record with probability, impact, response, owner, and status. |
| Issue register | Formal issue and change record. |
| Quality register | Planned and completed quality activities. |
| Lessons log | Lessons identified during the project. |
| Product register or product status account | Tracks product state, versions, and approval status where useful. |

## Reports

| Management Product | Purpose | Typical Timing |
|---|---|---|
| Highlight report | Project manager report to the project board. | During stage control |
| Checkpoint report | Team manager report to the project manager. | During product delivery |
| Exception report | Escalates forecast tolerance breach. | As needed |
| End stage report | Summarizes stage performance and supports next-stage authorization. | Stage boundary |
| End project report | Summarizes final performance, acceptance, open items, and follow-on actions. | Closing |
| Lessons report | Captures lessons for future projects. | Closing or stage boundary |

## Tailoring Rules

- Keep evidence for decisions, tolerances, acceptance, risks, issues, and exceptions.
- Combine artifacts when project size is small and governance allows it.
- Keep artifacts separate when different people approve, own, or audit them.
- Tool records can replace documents when they preserve the same evidence.
- Do not create documentation that nobody will use to make or audit a decision.
